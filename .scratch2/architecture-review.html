Architecture Review — Physics & Simulation Core

Your graph has loops. Your engine assumed it doesn't.

Why MuJoCo kept fighting you, what Blender/Unity/Gazebo/Isaac actually do under the hood, and a concrete plan for a physics core that treats an arbitrary body/joint graph — cycles included — as the normal case instead of a workaround.

Scope simulation & physics core only GUI/editor unaffected Status plan for review — nothing implemented
Direct answers first

Your code is not Java — it's TypeScript/JavaScript. Yes, C/C++ is possible and you already have a native C++ MuJoCo core sitting unused in native/. But a full rewrite is the wrong move: your GUI isn't the bottleneck, your physics engine's assumptions are. The fix is swapping the physics core to one built for graphs with loops — Jolt Physics — not rewriting the app.

01

What actually went wrong

Rapier wasn't broken. Reading your own codebase's history: the "shaking" problem you remembered was caused by a specific bad pattern — a chassis body floating with no ground contact of its own, held up entirely by wheel joints. That was diagnosed and fixed inside Rapier, before MuJoCo ever entered the picture. The fix (rigid clustering + proper multibody joints + real ground contact on every body) is documented in your own dynamicSim.ts history as already resolved.

MuJoCo was reached for on the belief that "more advanced" meant "more correct" everywhere. It's true for one narrow thing — pure joint-stability accuracy on a fixed tree — and false for almost everything else you actually need:

MuJoCo's actual design center mismatch

A research tool for training RL policies on a known, fixed, tree-shaped robot, run for millions of steps offline. It compiles one static model and reuses it forever. Great fit for that job.

What you're actually building the real spec

An interactive editor where the topology itself changes live — bodies, loops, arbitrary graphs, any number of wheels in any orientation — and has to feel instant while you build.

Every symptom you hit traces back to that mismatch: the duplicate-name crash (MJCF requires a strict tree; your data model doesn't), the UI freeze on load (MJCF compiles synchronously, blocking the one thread your UI also runs on), and the NaN spam (a workaround for tree-only structure left real overlaps unhandled). These weren't bugs to patch one at a time — they're the same root cause wearing different hats.

02

The actual fork: maximal vs. reduced coordinates

Every rigid-body engine picks one of two ways to represent a robot. This single choice is what determines whether "graph with loops" is your default case or your edge case.

REDUCED COORDINATES  (MuJoCo, DART, Drake, Simbody)
  Each joint removes degrees of freedom from a parent-child tree.
  Fast, numerically pristine for a chain — but a loop or a second
  parent isn't representable. It gets bolted on afterward as a
  soft "equality constraint," an approximation, not a real joint.

MAXIMAL COORDINATES   (Jolt, PhysX, Bullet, Rapier, Chrono)
  Every body is independently free (6 DOF); joints are just
  constraints between bodies. A loop is two more constraints —
  no special case, no approximation, no tree requirement at all.
  This is the family every general-purpose game/robotics engine
  outside pure RL research actually uses.

Your stated requirements — dynamic topology, closed loops, any wheel count/orientation, structures that aren't trees — only exist natively in the maximal-coordinate family. MuJoCo is, for your use case, the wrong family, not just a wrong implementation detail.

03

How the reference tools actually do it

You named Blender, Unity, Gazebo, and Isaac. Here's what's really under each hood, not the marketing version:

Gazebo graph-native

Describes robots in SDF, not URDF — URDF's spec explicitly forbids loops ("only tree structures... ruling out all parallel robots"). SDF references links by name with no tree requirement, so loops are first-class. Default physics is DART (reduced-coordinate, tree-based — loops via constraint), but Gazebo also ships a full Bullet backend specifically because DART's tree assumption doesn't cover everything.

Isaac Sim graph-native

PhysX 5 (maximal-coordinate, GPU-accelerated) plus USD for scene description. Every body is a real actor; joints are constraints. This is NVIDIA's answer to exactly your problem, at a scale far beyond what you need — worth knowing the pattern, not adopting the whole stack.

Blender different problem

Its dependency graph is a generic DAG for scene/animation evaluation (what depends on what to redraw), not a physics topology — it doesn't have your loop problem because it isn't solving your problem. Worth borrowing one idea: recompute only the subgraph that actually changed, not the whole scene, which is the fix for your "recompiles everything on every toggle" cost.

Unity DOTS/ECS different problem

Entities/components/systems is a data-layout pattern for CPU cache efficiency at scale (thousands of objects), separate from physics topology. Unity's own physics (Havok-based) is maximal-coordinate. Not something to adopt wholesale at your scale, but the "keep hot data flat and cache-friendly" principle is worth applying to whatever engine you land on.

05

Recommendation: Jolt Physics

Rapier would also genuinely work — it already did. Jolt is the recommendation over going back to Rapier for one reason: it's the only option in this table with real multithreading inside the WASM build itself (SharedArrayBuffer-backed worker threads), which directly attacks the "compiling/stepping blocks the UI thread" problem that's hit you twice now with MuJoCo. Rapier's WASM is single-threaded, same limitation MuJoCo has, just less severe. Jolt is also the physics engine actually shipping in current AAA titles at real scale (Horizon Forbidden West, Death Stranding 2) — proven, not experimental.

What moving to Jolt actually fixes

Loops and arbitrary graphs become the default case, not a special case you engineer around. Adding/removing a body or joint while the user drags things around is a cheap native operation again, like Rapier was — no MJCF recompile, no compile-on-every-toggle stall. Physics can run off the main thread, so a heavy step never blocks a click.

What happens to the MuJoCo work already built

Not wasted. native/tetrobot_core (real C++ MuJoCo, not WASM) stays as an optional offline training/export backend — exactly where MuJoCo's actual strength lives: batch RL training on a finished, fixed-topology design, disconnected from the live interactive editor. The editor and Jolt don't need it; a future "train this design" feature could.

06

Stack: C, C++, Rust, or stay TypeScript?

Your GUI (React, Three.js, the drag/drop editor) is not where your slowness comes from — it was never the bottleneck, and a benchmark or profile would show that clearly if it's ever in doubt. Rewriting it in C buys you nothing and costs months. The bottleneck is specifically the physics core's execution model.

LayerStay / changeWhy
Editor UI, panels, viewportStay TypeScript/React/Three.jsAlready works, not the bottleneck, huge rewrite risk for zero benefit
Model/graph data structureRestructure in placeNeeds to be a true graph (adjacency, cycles allowed) at the data-model level, not just patched at the physics-export step — see §7
Physics coreReplaceJolt via its official WASM build first; native C++ later only if profiling proves WASM is the ceiling

C specifically: possible, not advisable. Jolt and PhysX are C++, not C — there's no serious modern physics engine written in plain C anymore; the ecosystem moved on decades ago. If you ever do go native (Electron native addon instead of WASM), it'll be C++, and Jolt's own source is what you'd bind to — the exact code already proven in shipping games.

Rust: only relevant if you were starting from zero. Rapier is Rust, and it was already fine. Not a reason to switch again on top of switching engines.

   optional, offline only, not in this critical path:

The key shift from what you have now: the physics bridge becomes incremental (add/remove one body, don't rebuild the world) instead of compile-from-scratch (regenerate the whole XML document and reload it) every time anything changes. That single change is what makes interaction feel instant again.

08

Phased plan

1

Data model → true graph

Replace the "each body has one parent joint" assumption with real adjacency: a body can have any number of incoming/outgoing joint edges. This is the foundation everything else sits on — get it right once instead of patching around it per-engine like the last two rounds.

deliverable: graph data structure + cycle-safe traversal, unit tested against a deliberately looped fixture
2

Jolt WASM integration, one sim first

Stand up Jolt exactly where PhysicsSim lives today — same public API surface, internals swapped — so it's a controlled, testable replacement rather than a big-bang rewrite. Verify with real physics tests, the same discipline used throughout this project.

deliverable: PhysicsSim on Jolt, passing the existing determinism/motor/box-pool test suite
3

Incremental physics bridge

Replace "rebuild the whole world" with "diff the graph, apply just the changed bodies/joints to the live Jolt world." This is the piece that actually kills the lag — not a bigger engine, a smarter update path.

deliverable: adding/removing one body no longer triggers a full-model reload
4

Move physics off the main thread

Jolt's WASM build supports real worker-thread execution. Physics stepping happens off the thread your clicks and animations run on, so a heavy scene never freezes the UI again — structurally, not just via workarounds.

deliverable: gravity/drag interactions stay responsive under a large, physics-heavy model
5

Migrate the remaining sims

DynamicSim, RigidTumbleSim, BoxWorld, RL training — same pattern as phase 2, one at a time, each with its own passing test suite before moving to the next.

deliverable: nothing left depending on the MJCF-compile-per-toggle pattern
09

Decisions only you can make

Jolt now, or one more attempt at tuning MuJoCo first?
Recommended: Jolt. MuJoCo's blocking-compile behavior is architectural, not a tuning problem — no amount of parameter adjustment removes a synchronous WASM call on the main thread.
Keep the native MuJoCo C++ core at all?
Recommended: keep it, repurposed as an optional offline export/training path. It's already built and tested — deleting it recovers nothing.
Data model rewrite scope — minimal patch or real graph structure?
Recommended: real graph structure now. This is the third time a "just make the tree-walker not crash" patch has been needed. Fixing the actual data model once is less total work than a fourth patch later.
